Slide 1: Reward Function Design

We use a **hybrid reward** combining **immediate (dense)** signals with **delayed & sparse** rewards.

Reward Composition Summary


R(t) = immediate_tool_reward        ← dense, per action
     + world_tick_reward             ← delayed, per advance_week
     + invalid_action_penalty        ← dense, per failed action
     + win_bonus                     ← sparse, end of episode

Training: Discounted Returns + Advantage


# REINFORCE with discounted returns
G = 0
for t in range(T-1, -1, -1):
    G = reward[t] + γ * G        # γ = 0.99
    returns[t] = G

advantages = (returns - mean) / (std + ε)  # z-score normalisation
loss = -advantage * log_prob + kl_coeff * KL(π || π_ref)
  • **γ = 0.99** propagates long-horizon sparse rewards back to early actions
  • **KL penalty (0.05)** prevents policy from drifting too far from pre-trained model
  • **LoRA adapters only** — base model frozen, KL computed by toggling adapters on/off

    Why Both Delayed and Sparse Rewards?

    
Week 1: interview → hire → match     ← immediate: -$500, -$2K, +$1.8K
Week 2: advance_week                  ← delayed: +$1.8K billing, -$0 bench
  ...
Week 26: contract ends                ← delayed: candidate returns to bench
Week 52: episode ends                 ← sparse: win bonus if profit > $200K
  • **Dense rewards** (per tool call) teach the agent which actions are good/bad
  • **Delayed rewards** (`advance_week`) teach the agent the compounding effect of placements vs bench burn
  • **Sparse rewards** (win bonus) incentivise long-horizon planning — γ=0.99 propagates this back
  • Without dense rewards: agent gets no signal for 52 weeks → doesn't learn
  • Without delayed rewards: agent doesn't learn bench burn → over-hires
  • Without sparse rewards: agent doesn't optimise for episode-level profit
